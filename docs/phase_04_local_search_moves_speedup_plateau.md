# Phase 4.4.2 - Local-search speedup plateau

The local-search move benchmark shows a common GPU scaling pattern:

1. Small cases can favor the CPU because GPU launch/copy/reduction overhead dominates.
2. Medium cases improve quickly because the fixed GPU overhead is amortized.
3. Large cases plateau because both CPU and GPU time become approximately linear in the number of candidate moves.

The benchmark has one CUDA thread per move. Each move performs a small, fixed amount of work: check the proposed replacement resource, compute the new cost, compare with the old cost, and update aggregate counters. Once the GPU is saturated, adding more moves mostly adds more identical work. The GPU remains faster, but the speedup ratio stops increasing.

## How to read the plateau

Use these plots generated by `plot_local_search_moves_scaling.py`:

- `local_search_moves_speedup.png` - shows the speedup rising and then flattening.
- `local_search_moves_throughput.png` - shows CPU and GPU moves/ms. A flat GPU-throughput curve means the GPU reached a steady processing rate.
- `local_search_moves_gpu_breakdown.png` - separates kernel time from non-kernel overhead.

If GPU total throughput is almost constant from one large point to the next, then the speedup plateau is expected.

## Why the plateau is not a failure

The plateau means the benchmark found the useful GPU regime. For the measured data, the GPU is already processing far more moves per millisecond than the CPU, but it has reached a steady throughput. More moves increase runtime, but not necessarily the CPU/GPU ratio.

## How to push speedup higher

To increase speedup further, one of these must change:

- increase work per move, for example richer cost functions or route-local recomputation;
- keep task/resource/current-assignment data resident on the GPU across multiple local-search iterations;
- reduce host/device transfer and aggregate-output overhead;
- batch several neighborhood evaluations into one GPU call;
- optimize reductions for best move and violation counts.

This is a useful lecture point: GPU acceleration often improves until fixed overhead is amortized and then reaches a hardware/algorithm throughput ceiling.
